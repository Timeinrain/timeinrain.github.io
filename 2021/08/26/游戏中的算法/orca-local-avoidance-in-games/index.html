ORCA 与 RVO 的关系

ORCA，全称 Optimal Reciprocal Collision Avoidance，可以看作 RVO 思想的进一步整理和形式化。

RVO 提出了互惠避障：两个 agent 都为避免碰撞承担责任。ORCA 则把这种责任分摊转化成速度空间中的半平面约束，并通过求解一个低维线性规划来选择最接近期望速度的可行速度。

对于游戏来说，ORCA 的吸引力在于它给出了更清晰的工程接口：每个邻居生成一个约束，多个邻居的约束求交，最后选出离 preferred velocity 最近的速度。

半平面约束

假设 agent A 和 B 在未来某个时间范围内可能碰撞。ORCA 会计算一个最小修正向量，让相对速度离开危险区域。

由于避障责任由双方分担，A 只需要承担其中一半。于是对于 A 来说，B 会产生一条线性约束，合法速度位于某个半平面内。

多个邻居会产生多个半平面：

$$ v \in H_1 \cap H_2 \cap \cdots \cap H_k $$

目标是在这些约束中找到最接近期望速度 \(v_{pref}\) 的速度：

$$ \min ||v - v_{pref}|| $$

这让局部避障从“枚举很多候选速度”变成“求一个受约束的最优速度”。

游戏实现要点

ORCA 在游戏中通常作为局部避障层，放在导航路径和角色运动之间。

一个常见更新顺序是：

  1. 根据导航路径得到下一个路点。
  2. 计算朝向路点的期望速度。
  3. 查询邻近 agent 和静态障碍。
  4. 构造 ORCA 约束。
  5. 求解新速度。
  6. 将速度交给移动组件或动画系统。

为了让运动更像游戏角色，还需要在 ORCA 输出后加一些处理：

  • 限制转向角速度，避免瞬间转身。
  • 对速度做平滑，减少抖动。
  • 给重要角色更高优先级。
  • 在拥挤区域结合队形或行为树决策。

ORCA 的边界

ORCA 的理论模型通常把 agent 近似成圆，并假设每个 agent 都会按算法进行互惠避让。

游戏里这些假设未必完全成立。例如玩家角色可能不会配合，怪物可能有攻击状态，动画也可能限制位移。此时 ORCA 仍然可以提供一个安全速度建议，但不应该成为唯一决策。

另外，ORCA 是局部方法。它不会自动理解“换一条走廊更好”这类全局选择。遇到死锁或瓶颈时，仍然需要全局重规划、拥堵检测或高层行为控制。
