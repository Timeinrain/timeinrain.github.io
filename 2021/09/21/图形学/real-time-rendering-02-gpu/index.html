Overview

在早期，由于图像生成过程包含十分频繁的各种操作，GPU应运而生。

在速度上，GPU的计算速度是胜于CPU的。纵观GPU发展，其可编程性和可配置性都在逐步地提升，朝着Programmability和Flexibility的方向迈进，为图形编程人员提供了大量的可能。

GPU的速度优势主要来源于对大量高平行化任务的集中计算。例如有固定硬件支持z-buffer的实现，支持快速贴图存取及其他buffer的存取以及快速计算triangle traversal等功能。

Shader Core则是一些处理相对独立的微小处理器，例如坐标变换处理，颜色计算处理等。

但同时，延迟也是在设计处理器时不可缺少的考虑因素，处理数据的inout时将会造成处理器的卡滞，这将会对性能有一定影响。

数据并行架构

CPU拥有多个处理器，但通常按序列进行执行。为了降低延迟，许多CPU都拥有局部缓存，并拥有许多防止性能下降（stall）的多种技术手段。

然而GPU却多是拥有着极多的处理器，并被称作Shader Cores。由于数据的相似程度高，这些数据可以由GPU执行并行处理。

同时，这些指令调用又基本相互独立，并不十分需要共同的内存或者相互之间的数据共通。GPU对吞吐量进行优化，即数据处理的最大效率。由于没有设置足够的cache以及薄弱的逻辑控制，shader core的延迟总是比CPU来得严重。

但我们可以通过SIMD（Single Instruction Multiple Data）思想，来缓解latency。定义上来说，每一个pixel shader的调用指令可以被称作一个thread，不同于CPU中的thread，该thread由一小块包含输入的空间，以及执行所需计算需要的寄存器组成。多个使用同一shader program的thread汇聚成股，在NVIDIA中被称作warps，在AMD中被称作wavefronts。而这些warp执行的shader program采用lock-step的方式。由于执行相同指令，故当遇到内存存取时，所有的thread都同样面临该问题。避免停滞，此时GPU将会更换为另一组warp进行处理，而先前的warp继续存取内存。如此往复，当处理好最后一组warp的时候，最先的warp的存取已经结束，此时再换位第一组warp继续执行指令。

GPU管线

在先前引出的GPU的概念模型在具体实现时由不同的生产商自行决定实现方式，例如某些fixed-function的阶段有可能在添加一些附加指令之后变成了可配置等。

GPU实现了如上图显示的诸多功能，有些可配置，有些可编程，虚线标识的即为可选阶段。

Vertex Shader是完全可编程的，主要用于实现几何处理阶段。Tessellation和Geometry Shader都是可选择的阶段，但在某些GPU上是不支持的（例如某些移动端）。

Clipping阶段以及Triangle Setup & Triangle Traversal都是固定功能阶段在硬件层面实现的。

Screen Mapping则由窗口以及视角设置影响，是可配置的部分。

Pixel Shader则是完全可编程的，尽管Merger阶段并非可编程，但其也是高度可配置的。

Programmable Shader Stage

现代着色器程序使用的是统一着色器设计理念。所有的顶点、像素、几何等着色器使用共同的编程模型。

它们拥有一套相同的Instruction Set Architecture（ISA），实现该模型的处理器被称作Common-Shader Core，拥有这种Common-Shader Core的GPU则被称作拥有统一着色架构。用以根据不同的需求，由于都是使用Common-Shader Core，因此可以让GPU分配任务来均衡负载。

基本数据类型是32位单精度浮点型标向量。

draw call意为绘制一组图元的指令，将会使得调用并执行一系列绘制这组图元需要的shader。

每个可编程shader都拥有自己的两种输入：

  • uniform
    意为该输入在整个draw call过程中，不会有值的改变。
  • varying
    从三角形顶点或者光栅化中得到的输入。

shader虚拟机提供多种特殊寄存器来存储各种不同的输入输出。

可以看到，varying input寄存器的大小比constant input寄存器的大小要小得多，这是由于varying input需要根据每个顶点或者像素分别存储，而自然就有它们本身大小的约束，但在一整个draw call中，constant input总是多次被用到。

flow control 则是用于指令分支，主要与if以及case有关。Shader拥有与uniform输入相关的static flow control以及与varying输入相关的dynamic flow control。

可编程Shading及API的演变

Vertex Shader

顶点着色器是管线中的第一个着色器。

三角形mesh主要由在模型表面相互联系的顶点构成，除此之外还可传入可选属性例如三角形的颜色、uv坐标、表面法线等。

此处，三角形的表面法线并非由表面记录，而是记录在顶点上，而在渲染时，三角形mesh倾向于表示一个曲面，顶点法线仅用于表示该mesh的表面朝向，而非mesh本身的法线。

顶点着色器必须至少输出顶点的位置。

顶点着色器并不能创造或者销毁顶点，并且由一个顶点生成的数据也不能传递给其他顶点，每个顶点都是独立处理的，因此达到并行。

在vertex shader执行前通常会有输入聚合操作，实际上，获取数据并创建顶点在vertex shader层面实现，并对程序员不可见。

一般来说，vertex shader输出的数据将会被传输到下一个阶段进行光栅化再是片段处理，但一些GPU也有tessellation及geometry shader阶段，在此阶段对这些数据有进一步处理。

Tessellation Stage

Tessellation 阶段的存在允许我们渲染曲线表面。

例如此阶段，GPU将曲线的隐式描述式转化为一系列显示的三角形数据，毕竟输送曲面描述式的准确程度大于直接传输三角形数据。

同时，tessellation还能避免在某些情况下CPU与GPU之间的交流成为性能瓶颈，例如某些一直在变换形状的动画角色或物件。

Tessellation可以根据摄像机视角的参数来对特定曲面以精度来生成三角形，同时估计质量与效率，即LOD技术（Level Of Detail）。

Tessellation阶段通常由三种元素构成：

  • Hull Shader
  • Tessellator
  • Domain Shader

Hull Shader

Hull Shader主要有两个功能：

  • 告知Tessellator应该以什么配置生成多少三角形
  • 对控制点进行处理

同时，Hull shader也可选择对输入的表达式进行修改，或增加删除控制点等。然后Hull shader将会把control point数据与tessellation 控制数据一同输入到domain shader进行处理。

Tessellator

Tessellator是仅在tessellation中存在的固定功能阶段。

它主要根据hull shader给出的信息，为传入的数据增加顶点，以便后续domain shader进行处理。

信息包括：

  • tessellation实施的表面类型：triangle、quadrilateral或是isoline
  • tessellation factors

其中，isoline主用于渲染毛发。

tessellation factors主要控制细分程度，它由内部及外部factor组成。内部factor主要决定每个三角面或者四边面将会被细分多少次，而外部factor主要决定边缘将会被分割多少次。

通过允许分离控制，相邻的曲面边缘匹配，避免出现破面等现象。

hull shader也可以通过设定外部tessellation factor为0或NaN来使tessellator放弃该输入。

Domain Shader

domain shader则是利用先前传入的表面数据计算每个输出顶点的值，与vertex shader类似，domain shader的输入与输出一一对应，然后传递至管线之中。

尽管系统听起来十分复杂，但其旨在高效且解放每个shader的功能复杂度。

传递给hull shader的数据也经常不会经过任何二次修改，仅仅是简单地将其pass到下一个阶段。

同时，该shader也被用于根据估计距离及屏幕大小来优化渲染，例如在渲染某些平原表面时。

tessellator一般产生新顶点，告知其位置、组成的图元等数据。

domain shader利用生成的点的重心坐标经由其表达式来生成这些顶点的位置信息、法向信息以及uv等vertex数据。

Geometry Shader

几何着色器可以将图元转变成另一种图元，这是Tessellation阶段无法完成的。

例如将三角形通过创建line edge来以wireframe的形式渲染，然后再用总是面对摄像机的四边面代替line edge来模拟加粗的wireframe。

geometry shader的输入是单个object以及与其有联系的顶点。

这些object主要可能是一组三角形、线段或仅仅只是一个点。

例如，每个三角形可以多输入三个相邻点，每个折线即可以使用两个相邻顶点进行处理。在DX11 SM5.0的环境下，甚至可以输入高达32个控制点的patch。（原文中提到因此Tessellation stage 对于生成patch更有效率，不知为何）

几何着色器可以输出零或多个顶点数据，通常被认作是点、折线或三角形集。但几何着色器也可以什么都不生成，那么一个mesh就可以选择性地通过编辑顶点、增删图元来进行修改。

**Geometry Shader被设计为修饰输入数据或生成有限数目的数据复制。**例如，可以通过复制生成cube map的六个面，也可高效地生成级联阴影映射以及一些生成可变大小的粒子、生成毛发以及在阴影相关算法中寻找物体边缘等。

Geometry Shader也可以输出至多4个流数据，其中一个stream被传输至管线中，而其他的则可以输出到Render Target中用于其他用途。

geometry shader严格地保证输入输出地图元顺序，尽管这将会需要额外的顺序控制进而对性能有一定影响。

在draw call过程中，总共有三个部分可以进行工作创建：光栅化、细分阶段以及几何着色阶段。就资源及内存需求而言，集合着色器由于其高度可编程化，它的资源及内存需求用量是最不可预测的，因此并不十分鼓励使用，尤其涉及到移动端设备开发时。

Stream Output

标准状况下，GPU管线的数据传输从vertex shader到光栅化然后再是pixel shader，Stream Output的思想将在经过vertex shader 处理之后的顶点数据输出，不仅如此，光栅化阶段甚至可以完全被关闭，整个管线仅仅被用于一个非图形化功能的数据流处理器。

传回的数据带来了交互式处理的可能，这些操作在流水模拟及其他粒子系统效果中十分有用。

stream output传回的数据是浮点数的格式，因此会有一定程度的内存损耗。此外，stream output以图元为单位，而非顶点。如果此时输入一个mesh，在处理时多个三角形划分成多个顶点，三角形中共用点的信息将完全丢失，为此，更普遍的做法是将顶点作为图元集合输送。

stream output的输入输出顺序也是得到保证的。

Pixel Shader

在顶点着色、细分以及集合着色完成之后，图元经由裁剪阶段然后进行光栅化操作。此些阶段都是相对而言为fixed-function阶段，但同时给予了一定程度的配置性。

rasterizer则粗略地计算三角形以何种程度遮蔽某个像素格子。

部分或完全遮蔽某像素的三角形部分被称作为fragment，即片段。

三角形顶点存储的数据例如z-value等经由三角形表面在每个像素点上进行插值之后，被传递到pixel shader中进行处理。

三角形数据的插值类型是由pixel shader指定的，一般情况下使用透视正确的插值方法，也有其他诸如屏幕空间的插值方法。

在编程方面，经由插值的顶点数据的输出即是pixel shader的输入。此阶段，很多其他数据此时被暴露给GPU，例如screen position、可见性标志等。

通常情况下，pixel shader计算得到该片段的颜色值，产生一个不透明颜色值或者选择性修改深度值，然后在merge阶段再处理最终颜色值。在pixel shader阶段我们也可以修改光栅化阶段产生的z值。模板测试buffer则通常不可修改，但可以被传入到merge阶段，DX11.3已经支持修改该值。在SM4.0中，雾计算以及透明度测试阶段已经从merge阶段迁移到了pixel shader 阶段。

pixel shader同时也可以选择性不输出某些数据，例如clip plane：

最初，pixel shader仅仅能将输入传输到merge 阶段，但随着发展，越来越多的指令可以在此阶段执行。由此，***Multiple Render Targets(MRT)***的理念应运而生。

不同于直接将数据存储至颜色buffer或者z-buffer中，每个片段将可以生成多个不同类型的值输出到不同的buffer中，这些buffer被称作render target。

render target通常拥有x-axis以及y-axis，不同的API支持不同的大小，数量也是4-8个不等。

但尽管有着这些限制，MRT的功能依然十分强大。例如在一个pass中，可以选择输出color、物体标识以及世界坐标距离分别到不同的Render Target中。

MRT的产生同时带来了一种新的渲染管线的出现，即为deferred shading延迟渲染。

延迟渲染将可见性和着色分在不同的pass中处理。第一个pass将物体的位置材质存储到g-buffer中，第二个pass再进行光照以及其他效果的处理。

pixel shader的局限性在于，它仅可以写获得到的那个片段数据，并不能读取近邻像素的数据，因此在pixel shader阶段，某个fragment不能直接将它的数据输出给临近的片段使用，同样也无法读取到临近片段的数据。但尽管如此，依然有许多其他方法弥补该不足，例如在这个pass中输出的数据可以在下一个pass中进行使用，亦或者可以使用image processing的手段来进行图像层面的处理。

但也有例外，pixel shader可以间接地获取到临近片段的值。它们可以从梯度或者差分信息中直接获取到图像的临近片段变化这一信息，进而得知临近片段数据。

而这种信息是在插值过程中每一片段的值在临近沿着x、y坐标变化时变化中产生的。

（由于较为难以理解，放上原文：One is that the pixel shader can immediately access information for adjacent fragments during the computation of gradient or derivative information. The pixel shader is provided with the amounts by which any interpolated value changes per pixel along the x and y screen axes.）

这类信息在纹理映射部分将会起到较大作用，例如纹理过滤。

现代GPU通过使用2x2一组的fragment进行处理来实现以上特性。

unified core 拥有着这种存储相邻数据的空间，它利用一组warp中的不同thread来存储这类数据。

但也因此，这种梯度信息无法被有dynamic flow control的部分的shader获取到，因为在一组中的fragment必须都由共同的指令集处理，若有dynamic控制将会导致最后获取到的梯度信息无意义。

DX11通过引进Unordered Access View(UAV)来支持随机位置的写访问权限，OpenGL将其命名为Shader Storage Buffer Object(SSBO)。由于pixel shader并行且无序执行，这种UAV是在其间共享的。

同样，为了防止同时写的冲突操作造成数据丢失，GPU采用了原子操作定义的方式来避免同时写，但同时也带来了一定的阻滞（stall）可能。

但即便有了原子操作，依然无法避免处理的乱序，在某些透明度物体层序渲染时会造成不理想的结果。在标准管线中，这些fragment将被在merger阶段重新排序。DX11.3引进了*Rasterizer order views(ROVs)*来确保fragment的处理顺序。与UAV类似，它允许随机读写，但不同之处在于它严格按照恰当的顺序来组织数据。在此阶段，若以恰当的方式来进行blending操作，则在merge阶段便不需要再做任何事情了，但若检测到有乱序存取，那么pixel shader便只能处于忙闲等待其之前的三角形处理完毕。

Merging Stage

Merging Stage利用深度信息颜色信息等最终合并到frame buffer中作为最终的输出。

此阶段，模板测试以及深度测试将派上用场。若可见性测试通过，则进行blending操作，通常情况下，对于不透明物体仅仅是颜色的替代。而真正的blending则是对于一些半透明物体及合成效果而言的。

但考虑如下情况，在经历之前的各种shader操作之后发现该fragment其实并不可见，岂不是浪费了很多资源。因此，有许多GPU利用early-z方法来降低这种消耗。

在pixel shader执行之前，先利用z-buffer比对筛去无用fragment，以降低pixel shader的资源浪费的方法被称作early-z方法。但若pixel shader中有选择改变z值或者抛弃某个fragment的操作，那么此ealry-z方法将不再适用。

merging 阶段覆盖了从fixed-function阶段到fully-programmable的阶段，merging阶段尽管不可编程，但其是高度可配置的。

例如我们可以利用颜色属性以及alpha值通过各种加减乘除的组合甚至比特位逻辑操作来混合颜色，来达到例如遮罩、透明度叠加、减色、正片叠底等图像效果。

DX10允许两个颜色源与frame-buffer上的颜色进行blend，尽管无法与MRT适配。MRT有着其他的跨render target的blend手段。

Compute Shader

相比于实现图形效果，GPU亦可以用于非图形领域计算，典型的即为深度学习神经网络训练。

CUDA及OpenCL等平台将GPU用作一个强大的并行处理器。

Compute Shader被以线程组的方式执行以保证同一组内的所有threads是同步执行。

此外，由于CPU与GPU之间的交流不及GPU内部交流来得迅速，Compute Shader也有着性能方面的优势，因为其可以更加快速地获取到由GPU产生的数据，而不需要被GPU与CPU的交流延迟所影响。因此后处理效果也是Compute Shader的一个常用技术。

Compute Shader在粒子系统中也依然十分有用，Compute Shader也便利了一系列mesh层面的操作，例如面部表情动画、裁剪网格、图像滤波、深度精度、阴影、景深等。
