《Real-Time Rendering》读书笔记（一） 渲染管线

整体架构

渲染管线可以分为以下四个部分：

  • 渲染速度可以用 frames per second(FPS)来定义。
  • 采用流水线形式的渲染流程将由耗时最大的一个环节决定整体图像的渲染速度。因此，我们需要着重解决该瓶颈问题。

Application Stage

应用程序阶段，由于基本在CPU上执行，故用户对此阶段有着完全控制。因此，若在此阶段实施一些优化，对后续的执行效率也会有比较大的帮助。

不过，某些采用Compute Shader的工作也被称为Application Stage，与在CPU上执行不同，该shader在GPU上执行，并将GPU视作高并行能力的处理器。

例如，碰撞检测将会在此阶段进行，输入输出响应、加速算法等渲染管线无法直接很好地处理的过程都会在此阶段实现。

Geometry Processing

几何处理阶段主要分为以下四个substage：

Vertex Shading

顶点着色需要解决两个任务：

  • 顶点位置计算
  • 顶点表现数据

顶点位置计算

通常，在投影到屏幕的过程中，顶点将会被变换到很多个空间。一个模型最初会被保存在它本身的模型空间，不带有任何变换。

此时的坐标系统被称作model coordinates，也就是模型坐标系。当作用模型变换之后，此坐标系统变换到世界坐标系下，即world space。此world space是唯一的。顾名思义，所有经过world space transform的物体都会变换到世界坐标下，即都共处于此世界空间。

紧接着，由于所有的物体都需要被摄像机所看到，对于摄像机有自己的位置朝向等参数，故我们还需要将世界坐标下的物体经过view transform。此变换假设我们的摄像机处于原点，并以y轴正半轴为正上方，x轴正半轴为正右方，z轴负半轴为正前方。（不同API有不同的约定）此时的空间被称作相机空间。这些变换矩阵都由4x4矩阵表示。

顶点表现数据

为了渲染真实场景，我们也需要对顶点的外观有所处理，包括颜色材质及与灯光的作用等。

此过程更加通俗的翻译表述为shading，即着色。通过某个渲染的方程，在顶点层级或者像素层级进行着色处理。通常情况下，每个顶点存储大量与之有关的数据，例如位置信息、法向信息、颜色信息等。

在变换到view空间时，所有的object将会被缩放到一个被称作canonical view volume的unit cube中。

在GPU中由vertex shader执行的Projection操作是最先做的，包括了

  • orthographic（正交）
  • perspective（透视）

两种主要投影方式。

这两种投影都可以被4x4的矩阵描述，在投影之后即变换到了clip coordinates，即裁剪空间坐标。

在变换完成之后，z轴坐标将会被存在一个叫做z-buffer的缓存中。自此，一个三维模型便被投影到了二维空间中。

选择性顶点处理

在vertex processing之后，仍有几个可选过程，以如下顺序：

tessellation

对于一个物体，在渲染时需要兼顾性能与渲染质量，tessellation阶段将由hall shader, tessellator和domain shader组成。主要将几何图元进行细分或合并来进行性能或渲染质量上的平衡。

geometry shader

几何着色器获取从顶点着色器输出的图元数据，对其进行处理，它可以对这些数据进行减少或者抹去，亦可以将其增加细节，对其扩充。

stream output

在此阶段，我们可以将顶点数据选择性地输出出来以便我们后续pass中处理。例如在geometry shader中生成的粒子输出到cpu上进行碰撞检测等。

裁剪

针对那些部分处于视景体内的物体进行裁剪操作。

在经过坐标变换后，使用齐次坐标来执行裁剪操作。由于透视时坐标的变换往往不是线性的，故需要引入w作为辅助来确保准确。

最终将裁剪的图元准备好进行屏幕映射。

Screen Mapping

屏幕映射阶段即将裁剪所得数据通过图形API对屏幕空间产生一个映射关系。

对于OpenGL而言，z轴坐标范围为[-1,1]，左下角为(0,0)，而DirectX则是z轴范围为[0,1]，左上角为(0,0)。

Rasterization

光栅化阶段即将面片中的点转变成像素进行绘制。

主要分为两个阶段：

  • Triangle setup
  • triangle traversal

Triangle Setup

在此阶段，GPU将会计算三角形的微分、边的表达式等数据，将其组装。此阶段由固定功能硬件执行

Triangle Traversal

此阶段将会检测三角形所覆盖的像素，作为Pixel Processing的输入。

其中，可以在其中施加多种抗锯齿采样方法。

Pixel Processing

此阶段主要分为

  • Pixel Shading
  • Merging

Pixel Shading

此阶段是由可编程GPU核所控制的，因此，其支持用户自主编写pixel shader(亦被称作fragment shader)。

在此阶段可以对该像素实施大量操作，例如Texturing等。

Merging

此阶段主要处理现得出的当前像素与先前保存的同一位置像素的冲突。

此阶段并非完全可编程，但可通过对其进行一定的配置来达到诸多效果。

例如可见性处理，透明叠加处理等，通过综合运用以及对相应z-buffer,alpha channel以及配置stencil buffer等framebuffer来达到所预期的效果。

在通过这些阶段之后，这些像素点将准备最终输出到屏幕上。但为了避免用户直接看到被绘制的全部过程，通常情况下将会采用多缓冲的策略，通过交替刷新存储渲染结果的buffer来避免该问题。

Summary

通过四个主要流程：application, geometry, rasterization and pixel processing，物体从数据转变成了可视化的图像。

但同时，在离线渲染中也有不同的管线和渠道，例如电影生成的离线渲染一般采用micropolygon pipelines，或近来的ray tracing及path tracing。

同时，可编程化GPU的诞生及发展让渲染有了更多的个性化且更具特色。
