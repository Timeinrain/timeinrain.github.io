目标和边界

FFT 海水渲染的核心问题可以拆成两句话：

  • 用物理或统计模型生成“像海”的大范围波形。
  • 在实时渲染器里把这些波形稳定、可控、可优化地画出来。

Tessendorf 的 Simulating Ocean Water 和 Flügge 的 GPGPU FFT Ocean Simulation 都走的是频谱路线：先在频域里描述不同方向、不同波长的波有多少能量，再用 IFFT 变回空间域的高度、位移和法线。YivanLee 的“海洋模拟”系列更接近 UE 工程落点；NVIDIA OceanCS、GPU Gems、WebGPU/Unity 的 FFT 海水实现则可以从不同渲染 API 角度交叉验证资源组织、compute pass 和 shading 取舍。

这个系列默认使用的坐标约定是 UE 的世界空间：X/Y 为水平面，Z 为高度。频谱公式里常写的二维位置 \(\mathbf{x}\) 可以对应 UE 的 float2(WorldPosition.x, WorldPosition.y)，高度 \(h\) 对应 WorldPosition.z 偏移。水平位移会写入 X/Y，最终形成 choppy wave。

初版目标是：

  1. GPU 生成一组可循环的海面贴图。
  2. 网格顶点读取位移贴图完成海面起伏。
  3. 材质读取法线、泡沫、深度和反射信息完成水感。
  4. 调试视图能看到频谱、高度、位移、法线和最终 shading。

论文模型给了什么

频谱海面的输入不是一组手工波，而是一组统计参数：

  • 风向 \(\mathbf{w}\)。
  • 风速或主波长控制量。
  • 海面尺度 \(L\)。
  • 网格分辨率 \(N\)。
  • 能量系数 \(A\)。
  • 高频阻尼和短波限制。
  • 随机种子。

常用 Phillips 频谱可以写成：

$$ P_h(\mathbf{k}) = A \frac{\exp(-1/(kL)^2)}{k^4} |\hat{\mathbf{k}}\cdot\hat{\mathbf{w}}|^2 \exp(-k^2l^2) $$

实际实现时通常会调整阻尼项、方向项和短波衰减，但工程直觉不变：顺风方向的波更强，过长和过短的波都要被限制，中等尺度的波贡献海面主体形态。

有了频谱后，为每个频率点生成初始复振幅：

$$ \tilde{h}_0(\mathbf{k}) = \frac{1}{\sqrt{2}} (\xi_r+i\xi_i) \sqrt{P_h(\mathbf{k})} $$

其中 \(\xi_r\) 和 \(\xi_i\) 是高斯随机数。这个阶段通常只在初始化或参数变化时计算。运行时每帧更新的是：

$$ \tilde{h}(\mathbf{k},t)= \tilde{h}_0(\mathbf{k})e^{i\omega t} + \tilde{h}_0^*(-\mathbf{k})e^{-i\omega t} $$

深水波的色散关系常用：

$$ \omega(\mathbf{k})=\sqrt{g|\mathbf{k}|} $$

最后对 \(\tilde{h}(\mathbf{k},t)\) 做 IFFT 得到空间域高度 \(h(\mathbf{x},t)\)。

图源：Lutz Flügge, GPGPU FFT Ocean Simulation, Figure 4.1。

频谱海面的最小数据链

如果只做高度，海面会像上下起伏的布。实际海水还需要水平位移、斜率、法线和泡沫提示。因此初版建议从一开始就生成这些贴图：

资源 格式建议 内容 用途
H0Texture RGBA32F 或 buffer \(\tilde{h}_0(\mathbf{k})\) 和 \(\tilde{h}_0^*(-\mathbf{k})\) 初始化频谱
OmegaTexture R32F \(\omega(\mathbf{k})\) 时间演化
SpectrumTexture RGBA32F 当前帧频域高度/位移/斜率 IFFT 输入
DisplacementTexture RGBA16F xyz = 水平位移x, 水平位移y, 高度z 顶点位移和材质
NormalTexture RGBA16F 或 RG16F 法线或斜率 shading
JacobianTexture R16F 浪尖压缩程度 泡沫生成

位移的频域形式可以从高度频谱推导：

$$ \tilde{D}_x(\mathbf{k}) = -i\frac{k_x}{|\mathbf{k}|}\tilde{h}(\mathbf{k},t) $$

$$ \tilde{D}_y(\mathbf{k}) = -i\frac{k_y}{|\mathbf{k}|}\tilde{h}(\mathbf{k},t) $$

斜率可以写成：

$$ \tilde{s}_x(\mathbf{k}) = i k_x \tilde{h}(\mathbf{k},t) $$

$$ \tilde{s}_y(\mathbf{k}) = i k_y \tilde{h}(\mathbf{k},t) $$

这些频域量分别 IFFT 后，就能得到空间域的水平位移、斜率和法线。Flügge 的 GPGPU 论文重点展示了如何把这些计算放到 GPU 上，以避免 CPU 每帧搬运大块数据。

从频域到UE资源

UE 里的落地可以有两种写法：

  • UE4 风格：FGlobalShader + ENQUEUE_RENDER_COMMAND + RHI texture/UAV。
  • UE5 风格：FGlobalShader + RDG，也就是 FRDGBuilder、FRDGTextureRef、FComputeShaderUtils::AddPass。

如果项目版本允许，建议直接使用 RDG。原因是海水 FFT 天然是多 pass 管线：频谱更新、横向 FFT、纵向 FFT、结果重排、法线生成、泡沫更新，每一步都有中间资源。RDG 能比较清楚地描述资源读写关系，也更方便接 UE 后续的 render pass。

一帧里的工程流程

一帧中可以按这个顺序执行：

  1. 游戏线程更新海面参数：风向、浪高、时间、相机位置。
  2. 渲染线程把参数写入 uniform buffer 或 pass parameter。
  3. UpdateSpectrumCS 根据 H0、Omega 和时间生成当前频域数据。
  4. FFTCS 使用 ping-pong 纹理做横向 IFFT。
  5. FFTCS 再做纵向 IFFT。
  6. PostProcessOceanCS 做重排、归一化、位移打包、法线生成、Jacobian 计算。
  7. 海面 mesh 的 vertex shader 或 material world position offset 读取 DisplacementTexture。
  8. pixel shader 读取 NormalTexture、FoamTexture、SceneDepth、Reflection 等信息完成 shading。

这个流程里最容易踩坑的是同步和生命周期。不要在游戏线程读写 GPU 贴图，不要把 RDG 临时资源直接塞给材质跨帧使用。材质采样通常需要一个稳定的外部 render target 或 texture reference；RDG 中间结果要在合适时机导出。

初版效果需要哪些可视化

海水问题很容易“看起来不对，但不知道哪里错”。初版一定要做调试视图：

  • H0 频谱能量图：检查风向和能量分布。
  • 当前高度图：检查 IFFT 是否正确、是否棋盘翻转。
  • 位移图：检查水平位移方向和强度。
  • 法线图：检查斜率和坐标系。
  • Jacobian/泡沫图：检查浪尖区域是否稳定。
  • 最终水面：检查 tiling、闪烁、反射和远近过渡。

UE 中可以把这些贴图接到一个 debug material，或者写一个 editor viewport overlay。调试图越早做，后面越省时间。
