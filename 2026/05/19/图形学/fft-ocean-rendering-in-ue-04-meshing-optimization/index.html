FFT海水渲染在UE中的落地方案（四）：Projected Grid、LOD与无限海洋

基础网格能证明 FFT 海水管线成立，但它不是开放世界海洋的最终答案。本篇把 Projected Grid、LOD、无限海洋、远近海洋采用不同波的方案作为 meshing 优化项单独讨论。

基础网格的问题

第二篇中的规则平面有三个问题：

  • 顶点分布和屏幕无关：近处顶点不够，远处顶点浪费。
  • patch 有边界：相机飞远后容易看到平铺和尽头。
  • 波形尺度混在一起：近处需要真实位移，远处更需要稳定轮廓和低成本 shading。

因此 meshing 优化不只是“让网格更大”，而是让顶点预算集中到屏幕上真正需要的地方。

海洋网格的优化目标可以写成：

Projected Grid

Projected Grid 的思路是：不要在世界空间均匀铺网格，而是在屏幕或投影空间铺一张规则网格，再把它反投影到水面。这样顶点天然按照屏幕分布，近处密、远处疏。

基本流程：

  1. 在标准化屏幕空间生成一个 N x N 网格。
  2. 对每个网格点构造一条从相机出发的射线。
  3. 射线与平均水面平面 Z = OceanLevel 求交。
  4. 得到世界空间水面点。
  5. 再用 FFT 位移贴图对这个点做高度和水平偏移。

Projected Grid 的优势是屏幕分辨率利用率高，尤其适合大海平面。但它也有坑：

  • 相机接近平视水面时，射线与平面夹角小，远处点会拉得很长。
  • 相机低头看近处时，屏幕边缘可能没有有效交点。
  • 大浪高度超过平均水面时，单纯与平面求交会低估边界。

常见补救：

  • 对远处交点距离做 clamp。
  • 用保守水面高度范围扩大投影范围。
  • 对 horizon 区域单独加远景带。
  • 在相机穿过水面时切换 underwater 逻辑。

LOD环与Clipmap

另一条路线是 LOD ring 或 geometry clipmap。它在相机周围构建多圈网格：

每一圈都跟随相机移动，但按照自己的网格步长 snap：

LOD ring 的好处是稳定、好调、与 UE 的 mesh 提交流程更贴近。坏处是 ring 之间需要处理裂缝。常用方法：

  • skirt：在边界加垂直裙边，遮住裂缝。
  • stitch index：为不同 LOD 边界准备拼接索引。
  • morph：在过渡区域把高密顶点渐变到低密位置。
  • 材质淡化：用法线和泡沫遮掉轻微裂缝。

如果项目不是极端大海平视镜头，LOD ring 通常比 Projected Grid 更容易稳定上线。

UE Water 插件的实现更接近 quadtree/LOD ring 的路线。源码里 UWaterMeshComponent 暴露了 TileSize、TessellationFactor、FarDistanceMaterial、FarDistanceMeshExtent 等参数；FWaterQuadTree 负责 AddWaterTilesInsideBounds、BuildWaterTileInstanceData 和 AddFarMesh；FWaterMeshSceneProxy::GetDynamicMeshElements 再按材质和 density/LOD bucket 组织 draw call。

几个源码细节很值得自研 FFT 海水参考：

  • r.Water.WaterMesh.LODMorphEnabled 控制 LOD morph，关闭后更便宜但可能出现 popping。
  • FWaterQuadTree::GetLODDistance 用 pow(2, LOD + 1) * LODScale 定义 LOD 距离。
  • FViewWaterQuadTree::GetWaterLODParams 会根据观察点到水面高度的距离选择 LowestLOD 和 HeightLODFactor。
  • AddFarMesh 会为远处水面生成 8 个 quad 的 skirt，思路上对应“远海环带”。

这套结构说明，UE 官方水面并不是简单画一张大平面，而是把 tile、LOD、far mesh、材质分桶和可选 GPU quadtree 串起来。FFT 海面可以把位移贴图接入这种 mesh 框架，而不是从零写所有网格管理。

无限海洋

“无限”不是把 mesh 做到无限大，而是让玩家永远看不到边界。

基础策略：

  • 相机附近用真实网格。
  • 远处用低密度 ocean disk 或 horizon band。
  • 海面 UV 使用世界空间，不跟 mesh 边界绑定。
  • 网格中心跟随相机，位置做 snap。
  • 大世界项目启用 camera-relative rendering 或 world origin rebasing。

远景海天线可以单独处理：

远海不一定要真实顶点起伏。对于地平线，稳定的颜色、反射、雾和波纹方向往往比真实位移更重要。

如果沿用 UE Water 插件思路，可以把近中景交给 quadtree tile，把远景交给 FarDistanceMaterial 风格的远海材质。远海材质不采样完整位移，只采样低频法线、天空反射和雾化参数；这样能明显降低地平线闪烁。

远近海洋使用不同波

远近海洋采用不同波是一个非常实用的优化：

区域 几何 波形 Shading
近处 高密度网格 FFT 位移 + 交互波 完整法线、泡沫、折射
中景 中密度网格 低频 FFT 位移 简化泡沫、反射混合
远景 低密度网格或环带 低频高度或 Gerstner 天空反射 + 滚动法线

原因很直接：高频几何在远处会变成闪烁，既贵又不稳定。把高频从远处几何里拿掉，改用法线和 roughness 表现，画面反而更稳。

更好的方案是从源头拆频段。近景模拟保留高频，远景模拟只保留低频，避免 mip 后的随机闪烁。

UE中的落地顺序

建议不要一次把所有 meshing 优化都上完。落地顺序：

  1. 基础规则网格跟随相机。
  2. 加 snap，解决网格游泳。
  3. 加中远景 ring，解决边界。
  4. 拆近中远材质参数，降低远景位移。
  5. 加远海 horizon band。
  6. 如果镜头需要大范围平视，再换 Projected Grid。

调试指标

Meshing 优化要用指标检查，不要只凭感觉：

  • 屏幕上每个三角形平均覆盖像素数。
  • 海面顶点数量和 draw call 数。
  • FFT pass GPU 时间。
  • 海面材质 GPU 时间。
  • 地平线是否抖动。
  • LOD 边界是否出现裂缝。
  • 高速移动相机时海面是否游泳。

UE 的 stat gpu、profilegpu、RenderDoc 和 wireframe view 都要用起来。海水很容易被“效果好看”掩盖掉结构问题，直到进入大场景才暴露性能。
