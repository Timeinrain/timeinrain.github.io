如何使用这份索引

这篇文章不是单篇教程，而是一份 UE 游戏开发的技术点地图。它适合在三种场景下使用：

  • 准备学习路线时，看自己还缺哪一块。
  • 做项目拆解时，把需求落到 UE 的具体系统上。
  • 复盘问题时，快速定位应该查哪类文档、源码或示例项目。

UE 的技术点很多，真正重要的不是一次性全部背下来，而是知道它们分别解决什么问题、在哪个层级工作、和其他系统如何发生关系。

引擎基础

这一层决定了“UE 为什么这样组织项目”。如果这些概念不清楚，后面的 Gameplay、蓝图、资源管理和网络同步都会显得很碎。

  • 项目结构：uproject、uplugin、Source、Content、Config、Saved、Intermediate。
  • 模块系统：Runtime 模块、Editor 模块、Developer 模块、ThirdParty 模块。
  • 构建系统：UBT、UHT、Build.cs、Target.cs、模块依赖、预编译头、Unity Build。
  • 对象模型：UObject、AActor、UActorComponent、UStruct、UEnum。
  • 反射基础：UCLASS、USTRUCT、UPROPERTY、UFUNCTION、UINTERFACE。
  • 生命周期：Engine 启动、GameInstance 初始化、World 创建、Level 加载、BeginPlay、Tick、EndPlay。
  • 内存管理：GC、引用链、Outer、Package、CDO、默认对象、强引用、弱引用、软引用。
  • 配置系统：Default*.ini、Project Settings、Console Variable、Command Line。
  • 日志系统：UE_LOG、Log Category、输出窗口、日志文件。
  • 时间系统：World Time、Real Time、Delta Seconds、Timer、Latent Action。

需要重点理解的是：UE 不是“普通 C++ 程序加编辑器”，而是建立在反射、对象生命周期、资源序列化和模块加载之上的大型运行时。

C++、反射与蓝图

C++ 和蓝图不是互相替代的关系，更像是两个不同层级的生产工具：C++ 适合稳定、底层、可复用的系统，蓝图适合快速组合、可视化调参和内容侧迭代。

  • C++ 基础：头文件组织、模块 API 宏、前向声明、智能指针、容器、委托。
  • UE 容器：TArray、TMap、TSet、TQueue、TWeakObjectPtr、TSoftObjectPtr。
  • 委托系统：单播、多播、动态委托、事件分发器、生命周期解绑。
  • 反射暴露：属性编辑、蓝图可读写、分类、元数据、细节面板显示规则。
  • 蓝图类型：Blueprint Class、Level Blueprint、Blueprint Function Library、Macro Library、Interface。
  • 蓝图通信：直接引用、接口、事件分发器、组件消息、Subsystem。
  • 异步蓝图节点：Latent Action、Async Action、任务型节点。
  • 热更新与迭代：Live Coding、Hot Reload 的限制、编辑器重启边界。
  • C++ 与蓝图边界：用 C++ 定结构和协议，用蓝图配置表现和流程。

经验上，越靠近底层数据结构、网络协议和性能热点，越应该放在 C++；越靠近关卡配置、角色表现和内容调参，越适合留给蓝图。

Gameplay Framework 与 3C

3C 指 Character、Camera、Control，是游戏手感的核心。Gameplay Framework 则提供了 UE 对“游戏世界如何运行”的默认分层。

  • World 与 Level：世界上下文、Persistent Level、Streaming Level、World Partition。
  • Actor 系统：生成、销毁、Tick、Replication、组件组合、Actor 生命周期。
  • Component 系统：Scene Component、Actor Component、Child Actor Component、自定义组件。
  • GameMode：只在服务器存在，负责规则、出生、胜负、流程。
  • GameState：同步给客户端的全局游戏状态。
  • PlayerController：玩家输入、相机控制、客户端 RPC 入口、UI 入口。
  • PlayerState：玩家身份、分数、队伍、可跨 Pawn 保留的数据。
  • Pawn 与 Character：可被 Controller 控制的实体，Character 额外包含胶囊体和移动组件。
  • CharacterMovement：走、跑、跳、飞、游泳、网络移动预测、根运动协调。
  • Enhanced Input：Input Action、Input Mapping Context、触发器、修饰器、动态切换输入层。
  • Camera：Spring Arm、Camera Component、Camera Manager、镜头震动、镜头过渡。
  • Interaction：Overlap、Trace、Hit Result、Gameplay Tag 标记、交互接口。
  • Damage：Damage Type、Apply Damage、受击反馈、死亡流程。
  • Save Game：存档对象、槽位、序列化、版本兼容。

这一层的检查标准很直接：角色能不能被稳定控制，镜头是否舒服，交互是否清晰，规则是否放在正确的服务器或客户端对象上。

GAS 与战斗系统

GAS 是 Gameplay Ability System，适合处理技能、属性、Buff、冷却、消耗、预测和网络同步。它学习成本较高，但在复杂战斗或多人游戏中非常值得掌握。

  • Ability System Component：技能系统的核心挂载点。
  • Attribute Set：生命、法力、攻击、防御、移速等属性集合。
  • Gameplay Ability：技能逻辑、激活条件、取消、提交、任务。
  • Gameplay Effect：属性修改、持续效果、周期效果、Instant 效果。
  • Gameplay Cue：技能表现事件，例如命中特效、音效、屏幕反馈。
  • Gameplay Tag：状态、技能、阵营、免疫、打断条件的统一标记语言。
  • Ability Task：等待输入、等待动画、等待目标数据、等待事件。
  • Cost 与 Cooldown：消耗、冷却、公共冷却、可预测冷却。
  • Prediction：客户端预测、服务器确认、回滚与失败处理。
  • Replication Mode：Full、Mixed、Minimal 的同步成本差异。
  • Targeting：目标选择、范围检测、Trace、锁定、命中盒。
  • 战斗表现：受击停顿、击退、霸体、打断、连招、取消窗口。

如果项目只是简单交互，自己写轻量技能系统会更快；如果项目有大量技能、状态、Buff 和多人预测，GAS 的结构化收益会越来越明显。

AI 与寻路

AI 不是只有行为树。完整 AI 往往包含感知、决策、导航、动作执行、群体协作和调试工具。

  • Navigation：NavMesh、Recast、Detour、Nav Modifier、Nav Link、动态导航。
  • Path Following：路径请求、路径点、移动失败、到达判定。
  • AIController：控制非玩家角色，驱动感知、黑板和行为树。
  • Behavior Tree：Selector、Sequence、Task、Service、Decorator。
  • Blackboard：AI 的运行时记忆，用于保存目标、状态和决策变量。
  • EQS：环境查询，用于选择掩体、站位、目标点。
  • AI Perception：视觉、听觉、伤害感知、阵营感知。
  • StateTree：状态机式决策，适合清晰阶段和条件切换。
  • MassEntity / MassAI：面向大量实体的 ECS 风格框架。
  • Crowd：Detour Crowd、RVO Avoidance、局部避障。
  • 调试：AI Debug、NavMesh 可视化、行为树运行态、EQS 调试。

AI 的关键不是让每个节点复杂，而是让“感知、决策、执行”之间的数据边界清楚。

动画系统

动画系统连接了角色输入、移动、技能、物理和表现。只会做状态机还不够，复杂项目往往需要理解动画数据如何被混合、重定向和运行时修正。

  • Skeletal Mesh：骨架、蒙皮、Socket、Physics Asset。
  • Skeleton：骨骼层级、Retarget、IK Rig、IK Retargeter。
  • Animation Sequence：单段动画、压缩、曲线、Notify。
  • Animation Montage：技能、攻击、分段、Slot、Section、Root Motion。
  • Blend Space：速度、方向、姿态二维混合。
  • Animation Blueprint：Anim Graph、Event Graph、状态机、Layered Blend。
  • State Machine：状态、转移条件、同步组、过渡动画。
  • Aim Offset：瞄准偏移、上半身叠加。
  • IK：Two Bone IK、FABRIK、Full Body IK、Control Rig。
  • Motion Warping：攻击、跳跃、翻越时对 Root Motion 进行目标修正。
  • Control Rig：程序化骨骼控制、编辑器内绑定、动画修正。
  • Animation Notify：命中特效、音效、攻击判定、脚步事件。
  • Pose Cache 与 Linked Anim Graph：动画复用与性能控制。
  • Cloth 与 Ragdoll：布料、物理资产、死亡布娃娃。

动画的工程目标是让“状态正确”和“表现自然”同时成立：逻辑层不要被动画绑死，动画层也要能拿到足够信息。

UI 与交互

UE 的 UI 主要分为 UMG 和 Slate。UMG 面向游戏内界面与设计迭代，Slate 更接近底层和编辑器工具。

  • UMG：Widget Blueprint、UserWidget、Canvas、Overlay、Panel、ListView。
  • Slate：底层 UI 框架，常用于编辑器扩展和复杂工具。
  • CommonUI：跨平台输入、菜单栈、手柄导航、通用按钮体系。
  • MVVM：数据绑定、ViewModel、界面状态与数据状态分离。
  • UI 生命周期：Construct、PreConstruct、OnInitialized、Destruct。
  • 输入路由：鼠标、键盘、手柄、Focus、Input Mode、光标显示。
  • Drag & Drop：拖拽操作、背包、技能栏、编辑器工具。
  • UI 动画：Widget Animation、过渡、弹窗、提示。
  • 本地化：Text、String Table、Culture、字体回退。
  • 性能：Invalidation、Retainer Box、Slate Insights、Widget 重建次数。

UI 最常见的问题不是“做不出来”，而是状态来源混乱。推荐尽早决定：哪些数据来自服务器，哪些来自本地缓存，哪些只是表现状态。

渲染、材质与特效

渲染侧技术点很多，可以按“资产、材质、灯光、后处理、特效、底层扩展”来分。

  • Mesh：Static Mesh、Skeletal Mesh、Nanite、LOD、HLOD。
  • Material：材质节点、材质函数、材质实例、参数集合、Layered Material。
  • Texture：贴图压缩、Mip、Virtual Texture、Runtime Virtual Texture。
  • Lighting：动态光、静态光、Stationary Light、Lightmass、Lumen。
  • Shadow：Virtual Shadow Maps、Cascade Shadow Map、Contact Shadow。
  • Reflection：SSR、Reflection Capture、Lumen Reflection、Planar Reflection。
  • Post Process：Bloom、DOF、Tone Mapping、Color Grading、Custom Depth、Stencil。
  • Niagara：System、Emitter、Module、Parameter、Data Interface、GPU 粒子。
  • Decal：贴花材质、延迟贴花、投射范围、排序。
  • Rendering Feature：Forward、Deferred、Mobile Renderer、Path Tracing。
  • Shader：Global Shader、Material Shader、Compute Shader、Shader Permutation。
  • RHI 与 Render Thread：渲染线程、Render Dependency Graph、底层图形 API 抽象。

如果目标是游戏玩法开发，重点应放在材质实例、灯光、后处理和 Niagara 的使用；如果目标是技术美术或渲染程序，则需要进一步阅读渲染线程、RDG 和 Shader 扩展。

物理、碰撞与运动

UE5 使用 Chaos 作为核心物理系统。游戏开发中大多数物理问题，首先要看碰撞配置是否正确。

  • Collision：Object Channel、Trace Channel、Preset、Response。
  • Query：Line Trace、Sweep、Overlap、Multi Trace、复杂碰撞。
  • Physics Body：Simulate Physics、Mass、Damping、Constraint。
  • Physics Material：摩擦、弹性、物理表面类型。
  • Character Collision：胶囊体、地面检测、台阶高度、斜坡限制。
  • Constraint：铰链、弹簧、限制轴、断裂阈值。
  • Ragdoll：物理资产、碰撞体、约束、动画到物理切换。
  • Chaos Destruction：Geometry Collection、破碎、Field System。
  • Vehicle：Chaos Vehicle、轮胎、悬挂、驱动模型。
  • Projectile：Projectile Movement、弹道、反弹、命中回调。

碰撞问题排查时，优先检查 Channel、Preset、是否使用复杂碰撞、是否在服务器执行、是否被移动组件覆盖。

音频系统

音频经常被低估，但它对反馈和空间感非常关键。

  • Sound Wave：音频资产、压缩、流式加载。
  • Sound Cue：随机、混合、调制、条件播放。
  • MetaSounds：节点式程序音频与可参数化声音。
  • Attenuation：距离衰减、空间化、遮挡。
  • Concurrency：同时播放数量、优先级、抢占规则。
  • Submix：混音、效果链、录制、分析。
  • Quartz：节拍同步、音乐游戏或节奏系统。
  • Audio Component：跟随 Actor、淡入淡出、动态参数。

游戏音频的工程重点是“什么时候播、在哪里播、能否被打断、是否会叠爆”。

关卡、大世界与内容生产

这一层关注内容如何被组织、加载和协作生产。

  • Level：Persistent Level、Sub Level、Level Instance。
  • World Partition：网格化加载、运行时流送、One File Per Actor。
  • Data Layer：按任务、状态或版本组织世界内容。
  • Level Streaming：手动加载、蓝图加载、距离加载。
  • HLOD：远距离代理、合批、内存与绘制调用优化。
  • Landscape：地形、Layer、材质、草地、洞穴限制。
  • Foliage：植被实例、密度、剔除、交互。
  • PCG：程序化内容生成、规则图、地形与植被生成。
  • Spline：道路、轨道、河流、相机路径。
  • Sequencer：过场、镜头、动画轨、事件轨。
  • Take Recorder：录制角色、相机和实时表演。
  • Asset Pipeline：DCC 导入、FBX、glTF、重导入、命名规范。

大型项目里，关卡技术不只是美术问题，它会直接影响多人协作、加载时间、内存峰值和包体结构。

网络同步

UE 的网络模型以服务器权威为核心。网络同步的核心问题是：什么由服务器决定，什么由客户端预测，什么只需要本地表现。

  • 架构：Listen Server、Dedicated Server、Client、Authority、Autonomous Proxy、Simulated Proxy。
  • Actor Replication：bReplicates、Net Update Frequency、Relevancy、Dormancy。
  • Property Replication：Replicated、RepNotify、条件同步、Push Model。
  • RPC：Server、Client、NetMulticast、Reliable、Unreliable、Owner 要求。
  • Ownership：Actor Owner、Controller、连接归属、RPC 调用权限。
  • Movement Replication：CharacterMovement、移动预测、服务器校正。
  • Subobject Replication：组件、子对象、对象通道、同步生命周期。
  • Replication Graph：大规模 Actor 按空间或规则分发。
  • Iris：UE5 新一代复制系统，需要按项目版本确认成熟度。
  • Lag Compensation：延迟补偿、回滚检测、命中确认。
  • Prediction：输入预测、技能预测、表现预测、失败回滚。
  • Online Subsystem：登录、会话、好友、匹配、平台服务。
  • 网络调试：Net PktLag、Net PktLoss、Network Profiler、Insights。

网络系统最忌讳“先单机写完再补联机”。如果项目未来一定要多人，核心对象、技能、交互和状态从第一天就应该按服务器权威设计。

工具流与编辑器扩展

工具流决定团队效率。很多看似“开发慢”的问题，本质是缺少编辑器扩展、数据校验和自动化流程。

  • 插件开发：模块拆分、Runtime / Editor 分离、插件依赖。
  • Editor Utility Widget：快速做编辑器内工具。
  • Blutility：批处理资产、场景对象操作。
  • Python：资产导入、批量重命名、自动化处理。
  • Slate 工具：自定义窗口、Tab、面板、列表、属性编辑。
  • Details Customization：自定义细节面板。
  • Asset Action：资源右键菜单、批处理、导入导出。
  • Factory 与 Reimport：自定义资源类型、导入器、重导入逻辑。
  • Data Validation：资源检查、命名检查、提交前校验。
  • Commandlet：无界面命令行任务、CI、批量处理。
  • JSON / CSV / DataTable：外部数据接入。
  • 第三方库：ThirdParty 模块、静态库、动态库、平台差异。

当某个操作在团队里被重复做三次以上，就值得考虑做成工具。

构建、打包与平台

打包不是最后一步，它会反过来约束资源、模块、平台 API 和工程结构。

  • Cooking：资源烘焙、平台格式转换、引用收集。
  • Packaging：Pak、Io Store、压缩、加密、签名。
  • Build Configuration：Debug、DebugGame、Development、Shipping、Test。
  • Target：Editor、Game、Client、Server。
  • Dedicated Server：服务器目标、无渲染、部署脚本。
  • Patch 与 DLC：Chunk、Primary Asset Label、增量更新。
  • 平台差异：Windows、Linux、Android、iOS、Console。
  • 移动端：触控、包体、Shader 复杂度、纹理压缩、性能预算。
  • CI/CD：BuildGraph、RunUAT、自动构建、自动打包、自动上传。
  • 崩溃收集：Crash Reporter、符号文件、版本号。

建议在项目早期就建立最小打包链路。能在编辑器里跑，不代表能在目标平台稳定运行。

性能优化与调试

性能优化不能靠猜，先观测再下结论。

  • CPU Profiling：stat unit、stat game、Unreal Insights。
  • GPU Profiling：profilegpu、RenderDoc、GPU Visualizer。
  • 内存：Memory Insights、LLM、对象引用、资源常驻。
  • 加载：Async Loading、Asset Manager、软引用、流式关卡。
  • Tick 优化：禁用无用 Tick、Tick Interval、定时器、事件驱动。
  • Actor 数量：合批、对象池、Mass、组件拆分。
  • 渲染优化：LOD、HLOD、Nanite、实例化、遮挡剔除、材质复杂度。
  • UI 优化：Invalidation、Retainer、ListView 虚拟化、减少绑定。
  • 网络优化：Relevancy、Dormancy、Replication Graph、同步频率。
  • 动画优化：Update Rate Optimization、LOD、Pose Cache、骨骼数量。
  • 资源审计：Size Map、Reference Viewer、Asset Audit。
  • 调试工具：Gameplay Debugger、Visual Logger、Collision Analyzer、Console Command。

优化的基本顺序是：先确认瓶颈在 CPU、GPU、内存、IO 还是网络，再处理占比最高的那一项。

工程架构与长期维护

项目越大，越需要把“能跑”升级成“能改、能查、能扩展”。

  • 模块边界：核心运行时、玩法模块、表现模块、编辑器工具模块分离。
  • 插件化：通用系统做插件，项目专属逻辑留在 Game 模块。
  • Subsystem：GameInstanceSubsystem、WorldSubsystem、LocalPlayerSubsystem、EngineSubsystem。
  • 数据驱动：DataAsset、DataTable、Primary Asset、Gameplay Tag。
  • 事件系统：委托、消息总线、Gameplay Message Subsystem。
  • 状态机：角色状态、关卡状态、任务状态、UI 状态。
  • 存档架构：运行时状态、配置状态、账号状态分离。
  • 测试：Functional Test、Automation Test、Gauntlet。
  • 版本控制：Git / Perforce、LFS、大文件锁、二进制资产协作。
  • 命名规范：资源前缀、目录结构、蓝图变量、Tag 命名。
  • 代码规范：模块依赖方向、头文件依赖、日志分类、错误处理。

好的 UE 架构通常不是把所有东西抽象得很复杂，而是让每个系统知道自己的边界：谁拥有数据，谁负责同步，谁负责表现，谁可以被编辑器工具修改。

UE5 重点技术

UE5 相比 UE4，在内容生产、大世界、渲染和输入系统上变化很明显。学习时可以单独标记这些技术点。

  • Nanite：虚拟化几何，减少传统 LOD 制作压力，但需要理解材质、透明、变形等限制。
  • Lumen：动态全局光照与反射，适合实时迭代，但需要关注性能预算。
  • Virtual Shadow Maps：高质量动态阴影，常和 Nanite 配合。
  • World Partition：替代传统大世界手动分块的一套自动流送方案。
  • One File Per Actor：改善关卡多人协作，减少二进制资产冲突。
  • Data Layers：按状态、任务或版本组织世界内容。
  • Enhanced Input：更灵活的输入映射与运行时上下文切换。
  • CommonUI：跨平台 UI 输入和菜单体系。
  • Control Rig / IK Retargeter：更完整的动画绑定与重定向流程。
  • Motion Warping：让根运动更容易对齐目标点。
  • StateTree：行为逻辑和状态切换的新选择。
  • MassEntity：大规模实体模拟和数据导向框架。
  • PCG：程序化内容生成工具链。
  • MetaSounds：程序化音频和节点式声音逻辑。
  • Iris：新复制系统，适合关注 UE 网络同步演进。
  • Substrate：材质模型演进方向，需要按项目版本确认启用策略。

学习 UE5 时不要只追新功能。更重要的是判断：这个技术是否解决当前项目的真实问题，是否适合目标平台，团队是否有维护能力。
